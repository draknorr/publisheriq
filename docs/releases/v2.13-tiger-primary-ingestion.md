# v2.13 - Tiger Primary Ingestion

**Release Date:** May 1, 2026

Version 2.13 records the accepted and tested state after the Tiger/R2 primary-ingestion work landed. TigerData is now the primary target for accepted incoming ingestion and product-data writes, with R2 used for archived payload bodies/assets. Supabase is retained for auth/session, reference data, legacy compatibility, and product surfaces that have not been proven Tiger-backed.

This release note is intentionally narrow: it does not claim PICS has fully cut over, that Railway deployments/env cleanup are complete everywhere, or that every dashboard product surface now reads from Tiger.

## Highlights

### Tiger/R2 Primary For Accepted Incoming Data

- New product/ingestion writes should target TigerData, with R2 used for object/archive payloads where the Tiger path expects external storage.
- Supabase should not receive new product/ingestion writes unless a path is explicitly documented as a legacy/reference exception.
- Supabase remains valid for auth, sessions, credits, chat logs, reference/legacy tables, and product surfaces that are still Supabase/RPC-backed.

### Accepted PRs

- PR #8, `tiger-change-intel-r2-cutover`, fixed Tiger-primary change-intel worker startup and routed storefront inaccessible/no-data updates through the Tiger catalog writer in Tiger mode.
- PR #9, `tiger-review-claim-fix-20260501`, fixed ambiguous `appid` resolution in the Tiger review claim RPC and updated ingestion verification to match actual Tiger job names and GitHub gate variables.
- PR #10, `tiger-embedding-candidates-fix`, optimized Tiger embedding candidate selection, filtered non-embeddable game candidates, and added `max_batches` for capped manual embedding smoke runs.

### Embedding Smoke Evidence

- `embedding-sync.yml` manual workflow runs `25205576035`, `25205646372`, `25205710827`, and `25205779668` completed successfully on commit `7be82955` after PR #10 merged.
- Those runs verify the repaired Tiger embedding candidate path and the capped manual smoke mechanism. They should not be read as proof of a full embedding backfill or all-runtime cutover.

## Operational Guidance

- For new ingestion/product-data work, start from Tiger/R2 writer paths instead of adding Supabase writes.
- Before treating a dashboard/product surface as Tiger-backed, verify that its read path is actually served by `apps/query-api` or another documented Tiger contract.
- Keep Supabase credentials and clients where they are still required for auth/session/reference/legacy behavior.
